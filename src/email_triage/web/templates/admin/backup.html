Bundle = entire install. The full bundle contains every credential, audit row, and message-state row this install holds. Anyone with the bundle file and the passphrase has full access. Pick a strong passphrase (16+ random characters) and store the bundle apart from any other auth material.

Full bundle

Contains the database, the YAML config, optional TLS certs, and optionally the master key. Without the master key, the bundle's encrypted credential rows are unreadable -- which is the point if you also export a separate master key only bundle below.

Master key only

Tiny file (under one kilobyte) with just the encryption key for the credentials inside a full bundle. Store this somewhere different from the full bundle: different password manager entry, different cloud, different USB drive. Recovery requires both files plus both passphrases.

Restoring a bundle

Restore is CLI-only. On the target host, with the service stopped: 

email-triage restore <bundle>
# Auto-detects bundle type (full vs key-only).
# Prompts for the passphrase.
# Extracts to side-by-side .restore files.
# Print the swap-in instructions; operator runs them manually.

Don't drop a bundle into a running install. The CLI route forces a deliberate stop / restore / start dance.
