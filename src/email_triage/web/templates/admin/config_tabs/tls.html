TLS configuration

Pick the right approach for your install. Most deployments are well-served by the built-in ACME automation (free Let's Encrypt certs, hands-off renewal). Use the manual CSR path only when an institutional CA is signing on your behalf.

🤖 Auto (ACME) — recommended

Built-in ACME client speaks directly to Let's Encrypt (or any RFC 8555 directory). Publishes challenges via DNS-01 (RFC 2136) or HTTP-01. Writes cert + key into cert_dir; renews automatically on a 24-hour timer. Zero ongoing operator overhead.

Configure ACME
✍️ Manual CSR

Generate a Certificate Signing Request, hand it to your institutional CA, and import the signed certificate. Use when the CA is on the operator's side (corporate / university PKI) and you can't use ACME.

CSR / Manual import
🔗 Tailscale-issued LE

Tailscale exposes a tailscale cert command that fetches a Let's Encrypt cert valid for your Tailnet hostname. Useful when the install is Tailnet-only and DNS-01 isn't an option.

email-triage tls fetch-tailscale --hostname <host>

🔧 Self-signed — dev / fallback

Bootstrap-only fallback; browsers warn on every connect. WebAuthn won't work cross-host with self-signed. Generated automatically if cert_dir is empty at startup.

email-triage tls bootstrap
