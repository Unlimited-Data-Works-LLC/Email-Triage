Setup guides — Gmail + Office 365

Step-by-step walkthroughs for connecting Email Triage to a Gmail or Office 365 account. Each integration has TWO paths: a "push" path (real-time delivery — provider notifies Email Triage the instant new mail arrives) and a "poll" path (Email Triage asks the provider every N minutes). Push is faster + cheaper but requires more setup; poll is simpler + works everywhere.

Hobbyist note — connecting an Outlook.com / Hotmail.com account without an Azure tenant.

Microsoft killed the "register an app without a directory" path in 2024 (see walkthroughs 1-2). For a personal MS account, the cheapest path WITH a predictable cost ceiling is the $6/mo M365 Business Basic subscription. If that's not the right trade-off for you, Email Triage has a fifth path: add your Outlook.com / Hotmail.com mailbox as an IMAP account — bypassing Azure entirely. Push via IMAP IDLE, folder move, sent-mail mining all work. See walkthrough 5.

Jump to a walkthrough:
  1. Office 365 — push (Microsoft Graph subscription)
  2. Office 365 — poll (OAuth only)
  3. Gmail — push (Google Cloud Pub/Sub)
  4. Gmail — poll (OAuth only)
  5. Outlook.com / Hotmail via IMAP (no Azure tenant needed)

Before any walkthrough: have admin access to Email Triage's Integrations tab open in one browser tab; you'll paste values into it as you go.

1. Office 365 — push (Microsoft Graph subscription)

Real-time delivery via Microsoft Graph's /subscriptions endpoint. Microsoft posts a notification to Email Triage's webhook within seconds of new mail arriving. Email Triage then fetches the message via Graph.

You'll need:

  • Admin access to the Azure portal ( portal.azure.com) for the Microsoft account you want to read mail from.
  • A directory (tenant) to register the app in. As of 2024 Microsoft deprecated registering apps outside a directory; personal Outlook / Hotmail accounts no longer get a free path. You'll see this on Register an application:
    "The ability to create applications outside of a directory has been deprecated. You may get a new directory by joining the M365 Developer Program or signing up for Azure."
    Four ways to satisfy the gate (see Step 0 below; install-admin task, one-time). Recommended: paid M365 Business Basic ($6/mo) if you have a personal Outlook account + want a no-landmines tenant. Alternatives: existing org account, "Azure free" (real traps — read), or M365 Dev Program if you qualify.
  • Email Triage's public hostname (e.g. triage.example.com). Microsoft must be able to reach a webhook at https://<your-host>/webhooks/office365 over the public internet. LAN-only installs can't use push — fall back to poll (walkthrough 2).
  • A valid TLS certificate on Email Triage (Microsoft won't post to self-signed). Email Triage's TLS tab can mint one via Let's Encrypt if you don't already have one.

Step 0 — Get a directory (install admin only)

Microsoft requires you to register apps inside a directory (tenant). Org / workplace mail comes with one (yourcompany.onmicrosoft.com); personal Outlook.com / Hotmail.com accounts do NOT. If you see the "create apps outside a directory has been deprecated" error on App registrations → + New registration, you need a directory first.

Who actually needs to do this: the install admin (the operator running the Email Triage server) does Step 0 once. Once the app is registered + its credentials are pasted into Email Triage's Integrations tab, every end user can connect their own O365 account against the shared app via the OAuth device-code flow (Step 5). End users don't touch Azure Portal at all. So if you're adding YOUR account to someone else's Email Triage install — skip this whole step.

Three viable paths. Honest read on each:

Option A — Buy M365 Business Basic ($6/month, RECOMMENDED):

Most-honest path. Predictable $6/month flat, real tenant with full Exchange Online + Entra ID, no spending-limit dance, no 30-day clock. The cheapest "actually free of landmines" path for a personal-account install admin who wants O365 push working in production.

  1. Go to microsoft.com/microsoft-365-business-basic → buy with the personal Microsoft account you want as the install admin (or create a fresh one — your call).
  2. Microsoft provisions a tenant (youremail<hash>.onmicrosoft.com) during checkout. Skip the optional steps about adding users / setting up domains — you can do those later or never.
  3. Sign in to portal.azure.com with the same account → Switch directory → pick the new tenant. App registrations is available; continue with Step 1.

Option B — Existing org / school / association Microsoft account:

Zero cost if you already have one. Caveat: the app belongs to that org's tenant, so the org's admin can see it. Ask permission first. Some orgs block self-service app registration (admin must register on your behalf).

  1. Sign in to portal.azure.com with the org account.
  2. The org's directory is already there. Continue with Step 1.

Option C — Azure free sign-up (read the trap terms carefully):

Microsoft's "Azure free account" offer terms layer the trap in three pieces:

  1. 30-day promo credit ($200) expires. "Within 30 days of sign-up or upon exhaustion of the credits (whichever occurs first), you must upgrade to a pay-as-you-go subscription by removing the spending limit." Card is now live for charges.
  2. "12 months free" services revert to pay-as-you-go after 12 months. From Microsoft's terms: "For the services that are free for 12 months, any services you're using after 12 months has expired will continue to run and you'll be billed for them at pay-as-you-go rates."
  3. "Always-free" services ARE free at $0 within specified monthly quotas — but Microsoft controls the list + the quota tables, and can reclassify a service off the always-free list any time. App registrations + Microsoft Entra ID free tier sit here today; what they'll do tomorrow is Microsoft's call.

Net: this option leaves a card on file for an undefined future cost. Card + auto-billing means a mis-click in Azure Portal (or a Microsoft policy change you don't notice) can charge you. Most installs probably stay at $0 for years — but nobody outside Microsoft can promise it.

Pick this if you accept the uncertainty + watch Azure Cost Management alerts. The signup: azure.microsoft.com/free → Start free → personal MS account → identity + card verification ($1 temporary hold, reverses) → Azure provisions the directory → return to portal.azure.com → Switch directory.

Option D — M365 Developer Program (only if you qualify):

Microsoft tightened this in 2024. The "Join now" button now requires one of:

  • Active Visual Studio Enterprise subscription
  • Eligible Microsoft Partner Network membership
  • AI Cloud Partner Program enrolment

Without one of these you'll land on a "Welcome — you don't currently qualify for a sandbox subscription" page. If you have one: sign in at developer.microsoft.com/microsoft-365/dev-program, click Set up subscription, pick a region + admin username + password, wait ~5 min, then sign in to portal.azure.com as admin@<your-domain>.onmicrosoft.com + switch directory.

Honest TL;DR: Option A ($6/mo paid tenant) is the only path with a predictable cost ceiling for a personal-account install admin. Option B is free if you have an org account. Option C starts at $0 today but the future cost is whatever Microsoft decides + whatever you accidentally click on. Option D is the cleanest if you qualify; most operators don't.

After Step 0: you're signed in to Azure Portal inside a real directory. Continue with Step 1. The Tenant ID for this app will be the directory's GUID (visible in Microsoft Entra ID → Overview → Tenant ID), NOT the consumers / common literals from the pre-2024 path.

Step 1 — Register an app in Azure Portal

  1. Go to portal.azure.com → Microsoft Entra ID (formerly Azure AD) → App registrations → + New registration.
  2. Name it whatever you like ("Email Triage" works). Keep it short — this name appears on the consent screen when you authorize the app later.
  3. Supported account types:
    • If you'll only authenticate accounts from this directory (sandbox-tenant users, or your org's M365 users): pick "Accounts in this organizational directory only".
    • If you'll authenticate a personal Outlook.com / Hotmail.com account against this app (common when you set up the directory in Step 0 only to satisfy the gate + actually want to read your personal mailbox): pick "Accounts in any organizational directory and personal Microsoft accounts". This is the safe choice for most personal-account operators.
  4. Redirect URI: leave blank for now. We'll add it after we have the right value.
  5. Click Register.

After registration you land on the app's Overview page. Two values you'll copy from here:

  • Application (client) ID — the GUID at the top of Overview. This is the Client ID.
  • Directory (tenant) ID — the other GUID. This is the Tenant ID.
    • "Accounts in this organizational directory only": paste the GUID verbatim.
    • "Accounts in any directory and personal accounts": use the literal word common as the Tenant ID (NOT the GUID). This routes the auth through Microsoft's common endpoint so personal Outlook + org users can both sign in.
    Pre-2024 the "Personal Microsoft accounts only" option let you use the literal consumers Tenant ID without a directory. Microsoft deprecated that path; if you went through Step 0 you now have a directory + use common or the GUID instead.

Step 2 — Mint a client secret

  1. Inside the app: Certificates & secrets → Client secrets tab → + New client secret.
  2. Description: anything ("Email Triage primary"). Expires: pick 24 months (Azure caps at 24). Click Add.
  3. Copy the Value column immediately. Not the Secret ID — the Value. Azure hides it on next page load + there's no way to retrieve it. You'll paste this into Email Triage as the Client Secret.

Step 3 — Add Graph permissions

  1. Inside the app: API permissions → + Add a permission → Microsoft Graph → Delegated permissions.
  2. Tick these scopes:
    • offline_access — refresh token issuance (required; without it the app must re-auth every hour)
    • Mail.ReadWrite — read messages, mark read/unread, move, label, draft replies
    • Mail.Send — send draft replies + digests via SMTP-equivalent
    • User.Read — fetch the signed-in user's display name + email
    • Optional Calendars.ReadWrite — only if you want calendar features (free-slot finder, RSVP). Skip if mail-only.
  3. Click Add permissions.
  4. Org admin only: click Grant admin consent for <your tenant>. Personal account: skip — consent happens at first sign-in.

Step 4 — Paste into Email Triage

  1. Open Email Triage's Integrations tab → Office 365 OAuth section.
  2. Paste:
    • Client ID — from Step 1 (the Application (client) ID GUID)
    • Tenant ID — consumers, common, or the GUID, per Step 1
    • Client Secret — the Value from Step 2
  3. Click Save. The "Office 365 OAuth" card flips green.

Step 5 — Add the account + authorize

  1. Go to Add account → pick Office 365 as the provider.
  2. Enter the mailbox name (anything — "Work" works) + your O365 email address. Submit.
  3. Email Triage shows a device-code page: a short code + a URL (microsoft.com/devicelogin). Click the link, paste the code on Microsoft's page, sign in to your O365 account, approve the requested permissions (the scopes from Step 3).
  4. Return to Email Triage. Within ~30s the page advances — "✓ Authenticated as <your@email>".

Step 6 — Subscribe to push notifications

  1. Go to Integrations → Office 365 Push Subscriptions section.
  2. Find your account row → click Start push. Email Triage POSTs to Graph's /subscriptions endpoint asking Microsoft to deliver notifications to https://<your-host>/webhooks/office365.
  3. Microsoft validates the webhook (synchronous round-trip to confirm the endpoint responds). If your TLS cert is valid + the host is reachable, the row flips to "Active, expires <timestamp>".

Subscriptions expire in 71 hours (Graph maximum). Email Triage's renewer task refreshes every 24 hours automatically; no operator action needed.

Common failure modes

AADSTS50059 — "No tenant-identifying information found"

Post-2024: this almost always means the Tenant ID doesn't match the supported-account-types choice from Step 1. Common fix:

  • If app supports "any directory + personal accounts": Tenant ID should be the literal word common.
  • If app is "this directory only": Tenant ID should be the directory's GUID (visible in Azure Portal → Microsoft Entra ID → Overview → Tenant ID).
  • If you see this on a personal Outlook account + you skipped Step 0, you're hitting the deprecated "register outside a directory" path. Complete Step 0 (M365 Developer Program) first, then re-register the app inside the new tenant.
AADSTS65001 — "Admin consent required"

Org tenant requires an admin to consent to the permissions before any user can. Have an admin sign in to Azure Portal → your app → API permissions → Grant admin consent for <tenant>. Then retry the device-code flow.

AADSTS700016 — "Application not found in directory"

Client ID is wrong (typo when copying) OR the app is in a different tenant than the user. Re-verify the GUID on Azure Portal's app Overview page.

AADSTS9002313 — "invalid_request: token request body is malformed"

Usually a stale Client Secret. Mint a new one in Azure Portal (Step 2) + paste the new Value into Email Triage.

"Subscription validation failed" on Step 6

Microsoft couldn't reach Email Triage's webhook. Three causes:

  1. TLS cert not trusted — Microsoft rejects self-signed. Use Let's Encrypt via the TLS tab.
  2. Host not publicly reachable — push requires Microsoft → your install. If you're LAN-only, switch to poll (walkthrough 2).
  3. Firewall blocking inbound HTTPS — Microsoft's outbound IPs are documented at learn.microsoft.com. Open 443 inbound or skip push.

2. Office 365 — poll (OAuth only)

Simpler than push: Email Triage asks Graph every N minutes (configurable, default 5 min) "any new messages since the last poll?" No Azure subscription, no webhook reachability, no public hostname needed. Works for LAN-only installs + installs with self-signed TLS certs.

You'll need:

  • Same Azure app registration as push (Steps 1-3 of walkthrough 1). The poll path uses the same Client ID / Tenant ID / Client Secret.
  • Same Graph permissions (Step 3 of walkthrough 1).
  • No public hostname requirement.

Setup

  1. Complete Steps 1-5 of walkthrough 1 (Azure app + paste into Email Triage + add account + device-code auth).
  2. Skip Step 6. Don't start the subscription.
  3. Go to the account's edit page → Ingestion tab → tick Poll enabled. Set the interval (default 5 minutes — anywhere from 1 to 60 works; lower = fresher mail but more API calls).
  4. Click Save. The watcher starts within ~30s and begins polling.

Poll latency = the interval you set. New mail arriving 30s after a poll waits ~5 min (default) until the next tick. Push latency is ~5s. Pick poll when push isn't reachable.

3. Gmail — push (Google Cloud Pub/Sub)

Real-time delivery via Google Cloud Pub/Sub. Gmail publishes a notification to a Pub/Sub topic each time the user's mailbox changes; a Pub/Sub push subscription posts that to Email Triage's webhook. Email Triage then fetches the new message via the Gmail API.

You'll need:

  • A Google Cloud project (free tier is fine — Pub/Sub usage for one mailbox is well below the always-free allotment).
  • Email Triage's public hostname + valid TLS — same reachability requirement as O365 push. LAN-only ⇒ use poll (walkthrough 4).

Step 1 — Create a Google Cloud project (or reuse)

  1. Go to console.cloud.google.com.
  2. Top bar → project selector → New Project. Name it ("Email Triage"). Skip the org if you don't have one. Click Create.
  3. Note the Project ID (looks like email-triage-123456) — you'll paste it into Email Triage later.

Step 2 — Enable Gmail API + Pub/Sub API

  1. Nav: APIs & Services → Library.
  2. Search for "Gmail API" → click → Enable.
  3. Back to Library. Search for "Cloud Pub/Sub API" → click → Enable.

Step 3 — Configure OAuth consent

  1. Nav: APIs & Services → OAuth consent screen.
  2. User type: External (unless you have a Google Workspace org and want Internal). Click Create.
  3. App name: anything ("Email Triage"). User support email: yours. Developer contact: yours. Click Save and Continue.
  4. Scopes: click Add or Remove Scopes. Add:
    • https://www.googleapis.com/auth/gmail.modify (read mail, mark read, label, move, draft)
    • https://www.googleapis.com/auth/gmail.send (send drafts + digests)
    • https://www.googleapis.com/auth/userinfo.email (sign-in email)
    • Optional https://www.googleapis.com/auth/calendar (calendar features)
  5. Test users (External-mode only): add your Gmail address. Without this you'll hit "blocked: This app's request is invalid" at sign-in.
  6. Save.

Step 4 — Create OAuth client credentials

  1. Nav: APIs & Services → Credentials → + Create Credentials → OAuth client ID.
  2. Application type: Desktop app. Name: anything. Click Create.
  3. Copy the Client ID + Client secret from the popup.

Step 5 — Create a Pub/Sub topic + push subscription

  1. Nav: Pub/Sub → Topics → + Create Topic.
  2. Topic ID: email-triage-gmail (or anything; you'll paste this into Email Triage). Leave defaults. Create.
  3. Click the topic → Permissions → + Add Principal.
  4. Add principal: gmail-api-push@system.gserviceaccount.com (this is Google's Gmail Push service account — verbatim string, no typos). Role: Pub/Sub Publisher. Save. Without this, Gmail can't publish to your topic — most common silent failure.
  5. Back to the topic → + Create Subscription.
  6. Subscription ID: email-triage-push. Delivery type: Push. Endpoint URL: https://<your-host>/webhooks/gmail. Auth: Enable authentication → pick a service account that exists in your project (or create one with "Pub/Sub Publisher" role).
  7. Audience: leave blank (default uses the URL).
  8. Acknowledgement deadline: 60 seconds (default 10 is too tight for cold-start). Click Create.

Step 6 — Paste into Email Triage

  1. Open Email Triage's Integrations tab → Google OAuth section.
  2. Paste:
    • Client ID + Client Secret from Step 4
    • Pub/Sub topic name: projects/<your-project-id>/topics/<your-topic-name> — full path, not just the topic id.
  3. Save.

Step 7 — Add the account + start the watch

  1. Add account → Gmail as provider. Enter mailbox name + your Gmail address. Submit.
  2. Device-code page appears with the URL google.com/device + a short code. Sign in, paste the code, approve the scopes from Step 3.
  3. Back to Email Triage → "✓ Authenticated as <your@gmail>".
  4. Account edit page → Ingestion tab → tick Push enabled. Save. Email Triage calls users.watch on the Gmail API, telling Gmail to publish to your Pub/Sub topic.

Gmail watch expires every 7 days. Email Triage's renewer re-calls users.watch every 24 hours; no operator action needed.

Common failure modes

"Pub/Sub: User does not have permission to publish"

Forgot Step 5 #4 — add gmail-api-push@system.gserviceaccount.com as Pub/Sub Publisher on the topic. Without this, Gmail can't deliver notifications.

Webhook returns 401 / 403

The Pub/Sub push subscription is signing requests with a service-account-issued JWT; Email Triage verifies it. Mismatch causes (rare): wrong audience config in Step 5 #7, or the service account doesn't actually have roles/iam.serviceAccountTokenCreator on itself. See Google's Pub/Sub auth docs.

"This app is blocked" at sign-in (External mode)

Test user not added in Step 3 #5. Go back to OAuth consent → Test users → add your Gmail address.

Mail flowing in but Email Triage shows zero new messages

Test the subscription end-to-end: Pub/Sub topic → Subscriptions → Trigger → Pull a sample message manually. Look for the message ID + a payload with emailAddress + historyId. If empty, Gmail isn't publishing — verify the watch is active via users.getProfile from any Gmail API explorer.

4. Gmail — poll (OAuth only)

No Pub/Sub. Email Triage calls users.messages.list + users.history.list every N minutes. Simpler than push; works for LAN-only installs. The trade-off is latency (poll interval) + slightly higher Gmail API quota usage on busy mailboxes.

You'll need:

  • Steps 1-4 of walkthrough 3 (project, Gmail API enabled, OAuth consent, OAuth client). Skip Step 5 (no Pub/Sub topic needed).

Setup

  1. Complete walkthrough 3 Steps 1-4 (Google Cloud project, APIs enabled, OAuth consent, OAuth client credentials).
  2. Email Triage Integrations tab → Google OAuth: paste Client ID + Client Secret. Leave the Pub/Sub topic field blank.
  3. Add account → Gmail → device-code auth.
  4. Account edit → Ingestion → tick Poll enabled. Set interval (default 5 min). Save.

Push + poll are independent toggles. You can run both ("belt and suspenders" — if push delivery flakes, poll catches up). Most installs run one or the other.

5. Outlook.com / Hotmail via IMAP (no Azure tenant)

Microsoft killed the "register an app without a directory" path in 2024 + tightened the M365 Developer Program eligibility. For personal Outlook / Hotmail accounts without a paid M365 subscription, walkthroughs 1-2 (the full O365 / Graph path) require a $6/mo tenant or Azure's card-on-file gate.

Email Triage has a fallback that bypasses Azure entirely: add your Outlook.com / Hotmail.com mailbox as an IMAP account. Personal Outlook accounts still support IMAP + app passwords. Email Triage's IMAP code path is feature-complete for mail-read + actions; you keep most of what the full O365 path offers, give up some of the Graph-specific extras.

What you get vs. give up

Feature O365 (Graph) Outlook via IMAP
Real-time pushGraph subscriptionIMAP IDLE (also real-time)
Poll fallbackGraphIMAP poll
Folder move (actions)YesYes
Labels / tagsOutlook categoriesIMAP keywords (operator's IMAP keyword sweep covers this)
Draft repliesYes (Graph)Yes (IMAP APPEND to Drafts folder)
Sent-mail mining (style learning)YesYes (IMAP Sent folder)
Calendar (free-slot, RSVP)YesNo — IMAP has no calendar surface; use a Google calendar surrogate or skip calendar features
Rich attachment metadataGraph payloadIMAP FETCH (works, slightly thinner)

Step 1 — Enable 2FA + generate an app password

  1. Sign in to account.microsoft.com/security with your Outlook.com / Hotmail.com account.
  2. Two-step verification must be on. If it's off, Microsoft won't expose the App-passwords surface. Enable it first (works with any TOTP app or SMS).
  3. Once 2FA is on: Advanced security options → scroll to App passwords → Create a new app password.
  4. Microsoft generates a 16-character password. Copy it immediately — you can't view it again. Paste into a password manager.

Step 2 — Add the account in Email Triage

  1. Go to Add account. Provider type: IMAP (not Office 365).
  2. Fill in:
    • Display name: anything ("Outlook personal").
    • Email address: your full Outlook / Hotmail / Live address.
    • IMAP host: outlook.office365.com
    • IMAP port: 993
    • Use TLS: yes
    • Username: same as email address
    • Password: the 16-character app password from Step 1 (NOT your regular Outlook password — Microsoft rejects basic-auth with the regular password)
    • Sent folder: typically Sent. The Mine-now picker on /profile/style-data will surface the actual name if it's different.
    • SMTP (optional, if you want Email Triage to send digests / drafts back): host smtp.office365.com, port 587, STARTTLS, same username + app password.
  3. Save. Email Triage opens an IMAP connection within ~10s; the account row flips to "✓ Connected" on success.

Step 3 — Pick push or poll

  1. Account edit page → Ingestion tab.
  2. Push enabled: tick to run IMAP IDLE. Microsoft's IMAP server supports IDLE; new mail triggers a push from the server within ~5s.
  3. Poll enabled: tick to fall back to polling every N minutes (default 5). Safe alongside push as a belt-and-suspenders backstop.
  4. Save.

Common failure modes

"LOGIN failed" / "Authentication unsuccessful"

Microsoft rejects basic-auth with your regular Outlook password — modern auth only. You MUST use an app-password (Step 1). If app-passwords aren't visible under Advanced security options, enable two-step verification first; Microsoft hides the surface otherwise.

"App passwords" doesn't appear in account settings

Two-step verification isn't on yet. Enable it at account.microsoft.com/security → Two-step verification → On. App passwords then appears under Advanced security options.

Mail arrives but Email Triage shows nothing

IDLE may have stalled. Check Admin → Stats → Ingestion rollup; if uncovered is non-zero for IMAP, the watcher is failing. Restart the container OR toggle Push off + on in the account edit page to force a reconnect.

Some folders don't show up in the move-action picker

IMAP folder visibility lives in your Outlook web client settings — IMAP only sees folders flagged "show in IMAP". Open outlook.live.com → folder list → right-click the folder → "Show in IMAP". The next Email Triage folder refresh picks it up.
